# bioconda

Procedures for maintaining Acid Genomics R package recipes in
[bioconda-recipes](https://github.com/bioconda/bioconda-recipes).

## Recipe locations

All recipes live at `recipes/r-<pkgname>/meta.yaml` in bioconda-recipes.
Maintainer handles: `@acidgenomics`, `@mjsteinbaugh`.

## Global r-base pin

Bioconda's active r-base minor version is set in:
`bioconda-utils/bioconda_utils/bioconda_utils-conda_build_config.yaml`

```sh
# Check current pin
curl -fsSL https://raw.githubusercontent.com/bioconda/bioconda-utils/master/bioconda_utils/bioconda_utils-conda_build_config.yaml \
  | grep -A1 "r_base:"
```

All `noarch: generic` R recipes automatically pick up this pin on rebuild — no
recipe-side `r-base` version pin is needed. The build string (e.g. `r45hdfd78af_0`)
reflects which r-base minor was active when the build ran.

## Dependency solve failures (r-base migration)

When CI fails with `ExplainedDependencyNeedsBuildingError` / `Unsatisfiable dependencies`
involving `r-base`, the cause is almost always a **migration lag**: the global pin has
advanced but upstream deps haven't been rebuilt yet.

### Diagnosis

```sh
# 1. Confirm the global pin
curl -fsSL https://raw.githubusercontent.com/bioconda/bioconda-utils/master/bioconda_utils/bioconda_utils-conda_build_config.yaml \
  | grep -A1 "r_base:"

# 2. Check which r-base builds exist for each dep
curl -fsSL https://api.anaconda.org/package/bioconda/r-<pkg> \
  | python3 -c "
import sys, json
d = json.load(sys.stdin)
for f in sorted(d['files'], key=lambda x: x.get('upload_time',''))[-6:]:
    deps = [dep for dep in f['attrs'].get('depends',[]) if 'r-base' in dep]
    print(f['version'], f['attrs'].get('build'), deps)
"
```

Latest build string prefix tells you which r-base minor it was built against:
`r43` = 4.3, `r44` = 4.4, `r45` = 4.5, etc.

### Fix: build-number bump

To rebuild an existing version against the new r-base pin, bump `build: number`
by 1 in `meta.yaml` and open a PR. No other changes needed.

### Acid Genomics R dependency order (leaves → roots)

Must land in this order so each level's r45 builds are available before the next:

```
tier 1: r-goalie, r-acidgenerics        (no acid deps)
tier 2: r-acidcli, r-syntactic          (need r-goalie / r-acidgenerics)
tier 3: r-acidbase                      (needs tier 1 + tier 2)
tier 4: r-pipette                       (needs tier 1–3)
```

## `license_file` mismatch (LICENSE → LICENSE.md)

AGPL-3 packages use a plain `LICENSE` file; Apache-2.0 packages use `LICENSE.md`
(generated by `usethis::use_apache_license()`). When a package is relicensed or the
file is renamed upstream, the bioconda recipe must be updated to match.

CI failure signature:
```
ValueError: License file given in about/license_file (/opt/recipe/LICENSE)
does not exist in source root dir or in recipe root dir (with meta.yaml)
```

Fix: change `license_file: LICENSE` → `license_file: LICENSE.md` in the recipe.

### Check which file a release tarball contains

```sh
curl -fsSL https://github.com/acidgenomics/r-<pkg>/archive/v<ver>.tar.gz \
  | tar tz \
  | awk -F/ '{print $2}' \
  | grep -iE '^(LICENSE|LICENCE|COPYING)' \
  | sort -u
```

Run this for the **exact version pinned in the recipe**, not just the latest tag.
The rename may have happened between versions.

## Opening fix PRs via the GitHub Contents API

**Do NOT use `git push` for bioconda-recipes** — it is a ~700 MB repo and a
fresh fork push takes many minutes (even after `--unshallow`). Use the GitHub
Contents API instead; it makes changes in seconds with no pack upload.

### Workflow

```sh
# 1. Get base commit SHA
gh api repos/bioconda/bioconda-recipes/git/ref/heads/<base-branch> --jq '.object.sha'

# 2. Get current file SHA (needed for PUT)
gh api "repos/bioconda/bioconda-recipes/contents/recipes/r-<pkg>/meta.yaml?ref=<base-branch>" \
  --jq '.sha'

# 3. Create branch on fork from base SHA
gh api repos/mjsteinbaugh/bioconda-recipes/git/refs \
  --method POST \
  --field "ref=refs/heads/<new-branch>" \
  --field "sha=<base-sha>"

# 4. Commit file change (base64-encode the new content)
CONTENT=$(python3 -c "import base64; print(base64.b64encode(open('recipes/r-<pkg>/meta.yaml').read().encode()).decode())")
gh api repos/mjsteinbaugh/bioconda-recipes/contents/recipes/r-<pkg>/meta.yaml \
  --method PUT \
  --field "message=<commit message>" \
  --field "content=${CONTENT}" \
  --field "sha=<file-sha>" \
  --field "branch=<new-branch>" \
  --jq '.commit.sha'

# 5. Open PR
gh pr create \
  --repo bioconda/bioconda-recipes \
  --head "mjsteinbaugh:<new-branch>" \
  --base "<base-branch>" \
  --title "..." --body "..."
```

**Always target `master` as the PR base — never the bot's `bump/r_<pkg>` branch.**
Bioconda's lint CI only fetches `origin/master`; any other base causes:
```
git merge-base origin/bump/r_<pkg> HEAD
fatal: Not a valid object name origin/bump/r_<pkg>
```

To incorporate the bot's version bump, branch *from* `bump/r_<pkg>` (so the head
already contains both commits) but set `--base master` in the `gh pr create` call.
Close the bot's PR as superseded once the fix PR is open.

## Autobump PRs

bioconda's bot opens PRs named `bump/r_<pkgname>` when it detects a new upstream
tag. These PRs reset `build: number` to 0 and update the sha256. If CI fails on
an autobump PR, it is usually one of:

1. `license_file` mismatch (see above)
2. Dependency solve failure due to migration lag (see above)
3. Legitimate recipe issue (rare for noarch R packages)

Check for open autobump PRs:
```sh
gh pr list --repo bioconda/bioconda-recipes --state open \
  --search "r-<pkg> in:title" --json number,title,headRefName
```

## Verification

After PRs merge and builds publish (~10–30 min for bioconda CI):

```sh
# Confirm r45 build exists
curl -fsSL https://api.anaconda.org/package/bioconda/r-<pkg> \
  | python3 -c "
import sys, json
d = json.load(sys.stdin)
print([f['attrs']['build'] for f in d['files']][-4:])
"
# Expect: [..., 'r45hdfd78af_0']

# Local solve check
conda create -n test-r45 --dry-run -c bioconda -c conda-forge r-acidbase=0.7.6
```
